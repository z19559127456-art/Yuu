# Yu v0.1.6 — 修复群聊窗口切换后聊天记录清空

> **发布日期**：2026-05-07
> **git commit**：`39da462`

---

## 版本概述

v0.1.6 修复了群聊窗口切走再切回来后聊天记录被清空的问题。根因是 WebSocket 重连时强制切换到 1:1 对话 + 群聊消息从未持久化到数据库 + 前端没有加载群聊历史消息的机制。三个缺陷联动导致切换窗口后消息永久丢失。

---

## 修复内容

### 1. 群聊消息持久化到数据库

- `group_send` 处理器将用户消息和 Agent 回复写入 `messages` 表
- Message 模型新增 `group_id` 字段，区分群聊消息和单聊消息
- `database.py` 添加对现有数据库的自动迁移（`ALTER TABLE`）

### 2. WebSocket 重连时不再强制切换到 1:1 对话

- 添加 `initialSetupDoneRef` 标志，仅首次连接时自动创建对话
- 重连时跳过 `agent_list` → `create_conversation` → `setActiveGroupId(null)` 链条
- 群聊状态在重连后保持不变

### 3. 前端主动加载群聊历史消息

- 新增 `useEffect` 监听 `activeGroupId` 变化，发送 `get_group_messages` 请求
- 后端新增 `get_group_messages` WebSocket 处理，按 `group_id` 查询消息
- WebSocket `onopen` 中主动恢复当前会话/群聊的消息

### 4. 版本号统一

- `frontend/package.json`：0.1.5 → 0.1.6
- `backend/app/main.py`：0.1.4 → 0.1.6
- `backend/app/__init__.py`：0.1.2 → 0.1.6

## 根本原因

问题由 **3 个相互关联的缺陷** 共同导致：

### 缺陷 1：WebSocket 重连后强制切换到 1:1 对话

```
窗口切走 → WebSocket 断开 → 重连 → get_agents → agent_list 响应
  → 自动 create_conversation(第一个 Agent)
    → conversation_created 响应
      → setActiveGroupId(null)  // 清除群聊选中
      → setMessages([])          // 清除所有消息
```

`useWebSocket.ts` 中 `agent_list` 处理器在每次收到 Agent 列表时（含重连）都自动为第一个 Agent 创建对话，触发了 `setActiveGroupId(null)` + `setMessages([])`，直接清空群聊状态和消息。

### 缺陷 2：群聊消息从未持久化到数据库

`ws_handler.py` 中 `group_send` 只通过 WebSocket 实时推送消息到前端，完全没有调用 `db.add()` 写入数据库。消息仅存在于前端 Zustand 内存 store 中，一旦被清空就永久丢失。

### 缺陷 3：前端没有加载群聊历史消息的机制

- 没有 `useEffect` 监听 `activeGroupId` 变化来请求群聊消息
- 后端没有对应的 `get_group_messages` WebSocket 处理
- 对比 1:1 聊天有完整的 `get_messages` → `message_list` → `setMessages()` 链条

### 缺陷之间的联动

```
重连 → 切到 1:1 → 消息清空 → 切回群聊 → 无加载机制 → 空白聊天区
  ↑                  ↑                        ↑
缺陷 1            缺陷 2                   缺陷 3
```

三个缺陷串联形成完整的问题链路。修复必须同时解决三者才能彻底消除问题。
